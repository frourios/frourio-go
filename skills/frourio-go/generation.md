# Code Generation

frourio-go is itself a Go program. It scans `frourio.go` files, plus their
package declarations, and writes generated Go and JSON files.

## CLI

The binary has two subcommands:

```bash
frourio-go generate <api-dir> [--openapi <path>] [--template <path>]
frourio-go openapi  <api-dir> --output <path>    [--template <path>]
```

### `generate`

Produces:

- `<api-dir>/frourio_server.go` â€” `Handler()`, `Mount(mux)`, decode helpers
- `<route-dir>/frourio_relay.go` â€” per-route types and `DefineRoute`
- OpenAPI JSON at the path given by `--openapi <path>` (skipped when omitted)

Required: `<api-dir>` (the root of your `api/` tree).

Flags:

| Flag           | Meaning                                                       |
|----------------|---------------------------------------------------------------|
| `--openapi`    | Output path for OpenAPI JSON. When omitted, OpenAPI is not generated. |
| `--template`   | OpenAPI template (JSON) merged into the output. Default: `openapi_template.json` next to `--openapi`. Auto-created if absent and the path is the default. |
| `--watch`      | Reserved (returns "not implemented yet" today)                |

### `openapi`

Generates **only** the OpenAPI JSON, without touching Go files:

```bash
frourio-go openapi ./api --output ./docs/openapi.json
```

`--output` is required. Useful for CI doc-only updates or when you don't
want to disturb generated Go output.

## `go:generate` Convention

Place a single directive in the `main.go` (or any file) of the application:

```go
// tests/apps/basic/main.go
package main

//go:generate go run ../../.. generate ./api --openapi ./openapi.json

import (
    "log"
    "net/http"
    "time"

    "example.com/app/api"
)

func main() { /* ... */ }
```

In a real project that has frourio-go in its `tool` directive (Go 1.24+),
this becomes:

```go
//go:generate go tool frourio-go generate ./api
```

â€¦or just the pre-built binary on PATH.

Run with:

```bash
go generate ./...
```

## Generated Files

Two kinds of files are produced.

### Per-route: `frourio_relay.go`

In each directory that has a `frourio.go`, a sibling `frourio_relay.go`
appears. It declares:

- Per-method `Request` and `Response` types
- Per-status concrete response types (`GetStatus200`, `PostStatus404`, â€¦)
- The handler interface struct: `RouteHandlers`
- The middleware function types: `MiddlewareNext`, `<Method>Next`, etc.
- Context types: `MiddlewareAllContext`, `<Method>Context`, â€¦
- The `DefineRoute` constructor used by `route.go`

Header on every generated Go file: `// Code generated by frourio-go. DO NOT EDIT.`

### Root: `frourio_server.go`

In the api-root directory only. Declares:

- `Handler() http.Handler` â€” returns a fully wired `http.ServeMux`-based handler
- `Mount(mux *http.ServeMux)` â€” registers all routes onto an existing mux
- Wrapper functions for each (route, method) pair: decode â†’ validate â†’
  middleware chain â†’ handler â†’ response write
- A package-private `*validator.Validate` instance reused for all validation

### `openapi.json`

OpenAPI 3.0.3 JSON, indented. Written only when `--openapi <path>` (or
`openapi --output <path>`) is supplied â€” there is no default location.
Usually committed to source control to keep client generation reproducible.
See [openapi.md](openapi.md).

## Module / Import Resolution

The generator walks parent directories from `<api-dir>` to find the nearest
`go.mod` and parses the `module` directive. All cross-route imports use
import paths under that module.

If your repo has multiple modules (e.g., a vendored sample app in a sub-go.mod),
make sure the api-dir lives inside the desired module's tree.

## OpenAPI Client Generation

frourio-go treats OpenAPI as the standard client-generation boundary. For Go
clients, the `oapi-codegen` tool is wired up in this repo through a sibling
package and its own `go:generate` directive:

```go
// tests/apps/basic/openapiclient/generate.go
package openapiclient

//go:generate go tool oapi-codegen -generate types,client -package openapiclient -o client.gen.go ../openapi.json
```

Run in order:

```bash
go generate ./tests/apps/basic                  # frourio-go: regenerates relay/server + openapi.json
go generate ./tests/apps/basic/openapiclient    # oapi-codegen: regenerates client.gen.go
go test ./tests/apps/basic
```

The first command refreshes both the Go server and `openapi.json`. The second
turns the JSON into a typed Go client. The test then exercises the client
against a `httptest.Server` wrapping `api.Handler()`.

You can substitute any other generator that consumes OpenAPI 3.0 (TypeScript,
Python, Rust, â€¦). The output lives outside frourio-go's purview.

## Ordering and Idempotency

The generator sorts routes alphabetically by URL path before emission, so the
output is stable across runs. `git diff --exit-code` after `go generate ./...`
is a good CI check.

If the generator finds changes that would make existing handler signatures
invalid (e.g., a removed status code), the new relay file will compile but
the route file won't. Resolve by updating the handler.

## Recommended CI Checks

```bash
go mod tidy
git diff --exit-code -- go.mod go.sum

go generate ./tests/apps/basic
go generate ./tests/apps/basic/openapiclient
go generate ./tests/apps/middleware
git diff --exit-code -- tests/apps

go tool goimports -w main.go internal tests
git diff --exit-code -- main.go internal tests

go tool golangci-lint run ./...
go tool govulncheck ./...

go test ./...
```

The pattern: regenerate, then assert nothing changed in the working tree.
